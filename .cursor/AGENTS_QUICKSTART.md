# 🚀 Быстрый старт: Система агентов Cursor

## 📌 Что создано

Полная команда из 9 специализированных агентов для разработки:

1. **TeamLead** - координатор команды
2. **Architect** - архитектор
3. **Backend** - Python/API/БД
4. **Frontend** - React/UI
5. **DevOps** - деплой/CI/CD
6. **QA** - тесты
7. **Documentation** - документация
8. **Research** - исследования
9. **Refactoring** - оптимизация

## 🎯 Как пользоваться

### Базовое использование

Агенты активируются автоматически на основе контекста:
- Открываете `.py` файл → активируется **Backend Agent**
- Открываете `.tsx` файл → активируется **Frontend Agent**
- Открываете `.md` файл → активируется **Documentation Agent**

### Явный вызов агента

В любой момент можете явно указать какого агента вы хотите:

```
Задача для @backend: Добавь новый endpoint для получения статистики
```

```
@teamlead Мне нужно создать систему уведомлений с email и Telegram
```

```
@research Какой лучший способ хранить большие файлы?
```

## 💡 Примеры использования

### Пример 1: Простая задача
```
@backend Добавь валидацию email в модель User
```
Backend агент сразу поймет что делать и сделает.

### Пример 2: Средняя задача
```
@research Какую библиотеку использовать для генерации PDF?
  ↓
Research исследует варианты и предложит решение
  ↓
@backend Реализуй экспорт отчетов в PDF используя WeasyPrint
  ↓
Backend реализует фичу
  ↓
@qa Напиши тесты для PDF экспорта
  ↓
QA добавит тесты
  ↓
@docs Добавь инструкцию по экспорту в README
```

### Пример 3: Сложная задача
```
@teamlead Создай систему подписок с оплатой через Stripe

TeamLead проанализирует и создаст план:

План:
1. @research → Изучить Stripe API и best practices
2. @architect → Спроектировать схему БД и API
3. @backend → Реализовать:
   - Модели БД
   - API endpoints
   - Интеграцию со Stripe
   - Webhook обработчик
4. @frontend → Создать:
   - Страницу выбора плана
   - Форму оплаты
   - Страницу управления подпиской
5. @devops → Настроить:
   - Webhook endpoint в nginx
   - Переменные окружения для Stripe
6. @qa → Написать тесты:
   - Unit тесты для сервисов
   - Интеграционные для API
   - E2E для оплаты
7. @docs → Документировать:
   - API endpoints
   - Процесс интеграции
   - Troubleshooting
```

## 🎓 Best Practices

### 1. Начинайте с TeamLead для сложных задач
```
✅ ХОРОШО:
@teamlead Создай фичу экспорта данных

❌ ПЛОХО (для сложной задачи):
@backend Сделай экспорт
@frontend Добавь кнопку
@qa Напиши тесты
... (вы сами координируете)
```

### 2. Research перед реализацией
```
✅ ХОРОШО:
@research Как лучше хранить файлы для Telegram бота?
  ↓ получаем рекомендации
@backend Реализуй хранение файлов в S3

❌ ПЛОХО:
@backend Реализуй хранение файлов
... (без исследования можем выбрать не лучшее решение)
```

### 3. Architect для новых фич
```
✅ ХОРОШО:
@architect Спроектируй систему подписок
  ↓ получаем архитектуру
@backend Реализуй по плану

❌ ПЛОХО:
@backend Сделай подписки
... (без архитектуры может быть плохой дизайн)
```

### 4. QA после изменений
```
✅ ХОРОШО:
@backend Добавил новый endpoint
@qa Напиши тесты для нового endpoint

❌ ПЛОХО:
@backend Добавил endpoint
... (без тестов)
```

### 5. Documentation в конце
```
✅ ХОРОШО:
[Реализована фича]
@docs Задокументируй новый endpoint в API.md

❌ ПЛОХО:
[Реализована фича]
... (без документации никто не знает как использовать)
```

## 🔄 Workflow примеры

### Sprint Planning Workflow
```
День 1: Планирование
@teamlead У меня 3 задачи на неделю:
- Добавить экспорт в Excel
- Исправить баг с webhook
- Обновить UI дашборда

TeamLead создаст план и приоритеты

День 2-6: Реализация
[Работа с конкретными агентами по плану]

День 7: Ревью
@architect Ревью новой архитектуры
@qa Проверь что все тесты проходят
@docs Проверь что документация обновлена
```

### Bug Fix Workflow
```
1. Обнаружение
@backend В логах ошибка: [лог ошибки]

2. Анализ
Backend анализирует, находит проблему

3. Исправление
Backend исправляет

4. Тестирование
@qa Добавь тест для этого edge case

5. Деплой
@devops Задеплой исправление на production
```

### Feature Development Workflow
```
1. Исследование
@research Как реализовать real-time уведомления?

2. Архитектура
@architect Спроектируй систему уведомлений

3. Разработка (параллельно)
@backend Реализуй WebSocket сервер
@frontend Добавь WebSocket клиент

4. Интеграция
[Backend + Frontend интегрируются]

5. Тестирование
@qa Напиши E2E тесты для уведомлений

6. Инфраструктура
@devops Настрой WebSocket в nginx

7. Документация
@docs Задокументируй API уведомлений
```

После создания системы агентов, проверьте:

```bash
ls -la .cursor/rules/agents/
# Должно быть 9 файлов: teamlead.mdc, architect.mdc, ...
```

### 2. Проверить AGENTS.md
```bash

```

### 3. Тестовая задача для Backend
```
@backend Создай простую функцию для валидации email
```

### 4. Тестовая задача для TeamLead
```
@teamlead Создай план для добавления системы логирования
```

## ⚡ Продвинутое использование

### Параллельная работа
```

TeamLead запустит агентов параллельно:
├─> @refactor Оптимизируй backend (async)
├─> @frontend Оптимизируй UI (async)
└─> @devops Настрой кэширование (async)

После завершения всех:

  ↓

  ↓
Architect создаст план
  ↓
@backend Реализуй по плану
  ↓
Backend реализует
  ↓

  ↓
DevOps настроит
  ↓

  ↓

```

## 🐛 Troubleshooting

### Агент не активируется
- Проверьте что файл правила существует в `.cursor/rules/agents/`
- Проверьте формат `.mdc` файла (frontmatter должен быть корректный)

### Агент делает не то
- Уточните задачу более детально
- Явно укажите что именно нужно сделать
- Дайте контекст (какой файл, какой проект)

### Нужен агент которого нет
- Создайте новое правило в `.cursor/rules/agents/`
- Используйте существующие правила как шаблон
- Добавьте описание в AGENTS.md
